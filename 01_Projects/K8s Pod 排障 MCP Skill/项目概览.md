# K8s Pod 排障 MCP Skill 项目概览

背景：

希望通过 `MCP + Skill` 的方式，让 Codex 使用 `kubectl` 连接 Kubernetes 集群，查看 Pod 信息，并辅助排查线上或测试环境问题。用户当前不希望写代码，因此项目第一阶段以配置、权限边界和 Skill SOP 为核心。

问题：

需要让 Agent 能够在多环境、多部署地点、多业务线中准确选择集群，并在不执行变更操作的前提下完成 Pod 运行状态和常见故障诊断。

方案：

采用 `kubectl + 只读 kubeconfig + MCP 命令白名单 + Skill 排障流程` 的组合：

- `kubectl` 负责实际访问 Kubernetes API。
- 只读 kubeconfig 和 Kubernetes RBAC 提供硬权限控制。
- MCP 工具层限制允许执行的 kubectl 命令。
- Skill 负责环境解析、目标发现、诊断流程、输出格式和禁止事项。

为什么这么做：

这种方式不需要立即开发完整 MCP 服务，落地成本低，同时能把 Kubernetes 排障经验沉淀为可复用 Skill。真正的线上安全边界由 RBAC 和 MCP 白名单承担，Skill 只作为模型行为规程，不承担硬权限职责。

踩坑：

- 不能把 Skill 当作安全边界。
- 不能默认使用当前 kubeconfig context。
- 不能默认查询生产环境。
- 不能默认业务名就是 namespace。
- 不能直接输出大段日志。
- 不能读取 Secret 或执行变更操作。

下次怎么复用：

遇到“查看某环境下某服务 Pod 运行情况”或“排查 Kubernetes Pod 异常”时，按以下流程复用：

1. 解析 `env:region:biz` 或 `env_region_biz`。
2. 选择只读 kubeconfig。
3. 按 namespace -> workload -> service -> label 发现目标。
4. 采集 Pod、describe、event、logs、resource 证据。
5. 匹配 CrashLoopBackOff、ImagePullBackOff、OOMKilled、Pending、Ready 异常等规则。
6. 输出结论、证据、影响范围、可能原因、建议下一步、风险、待确认项。

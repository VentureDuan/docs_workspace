

**调查结论**

此次 `cgbundle` 服务组分发变慢，根因不是业务逻辑慢，而是 Temporal 在 YL 环境触发了 namespace 级别的 poller 并发限制：

```text
namespace concurrent poller limit exceeded
Failed to poll for task.
```

这会导致 `WorkflowWorker` / `ActivityWorker` 无法及时从 Temporal 拉取任务，表现为 activity 已完成，但 workflow 很久后才进入下一步。

**关键证据**

1. 新增日志后确认 `UninstallApps` 很快：

```text
UninstallApps cost=4.38s
```

2. 多个阶段耗时异常，但业务日志显示 activity 实际已较早完成，延迟主要发生在 Temporal worker 获取/处理下一次任务阶段。

3. `cginstanceagent` 日志中大量出现：

```text
WorkflowWorker namespace concurrent poller limit exceeded
ActivityWorker namespace concurrent poller limit exceeded
```

**环境差异**

prod 环境对 `cgbundle` 配置了较高的 namespace poller 限制：

```yaml
frontend.namespaceCount:
  - constraints: { namespace: 'cgbundle' }
    value: 100000
```

YL 环境只给 `default` namespace 配置了该限制，`cgbundle` 未配置，因此走 Temporal 默认值，导致更容易触发限制：

```yaml
frontend.namespaceCount:
- constraints:
    namespace: default
  value: 100000
```

**根因**

YL 环境 `cgbundle` namespace 缺少 `frontend.namespaceCount` 等动态配置，Temporal frontend 对该 namespace 的并发 poller 限制过低。`cginstanceagent` 实例数较多、每实例都有 workflow/activity worker，累计 poller 数超过限制后，任务拉取被拒绝，造成 workflow 推进延迟。

**处理后的结果**

1. 将 YL 环境 `frontend.namespaceCount` 补齐到 `cgbundle`，与 prod 保持一致：

```yaml
frontend.namespaceCount:
- constraints:
    namespace: cgbundle
  value: 100000
- constraints:
    namespace: default
  value: 100000
```

2. 验证服务分发速度，任务速度由原来的 20分钟，降低到30s

3. 变更后观察 `cginstanceagent` 日志中没有出现：

```text
namespace concurrent poller limit exceeded
```



`cgbundle` 服务组分发变慢问题已定位，结论如下：

【问题现象】  
服务组分发过程中，`UninstallApps` 已执行完成，但很久后才进入 `InstallApps`，导致整体 workflow 推进变慢。

【根因】  
不是业务逻辑慢，而是 YL 环境 Temporal 触发了 namespace 级 poller 并发限制：

```text
namespace concurrent poller limit exceeded
Failed to poll for task.
```

该限制会导致 `WorkflowWorker` / `ActivityWorker` 无法及时从 Temporal 拉取任务，所以 activity 完成后，workflow 不能及时推进到下一步。

【关键证据】  
新增日志后确认：

```text
UninstallApps cost=4.38s
```

说明卸载本身很快。

同时日志中大量出现：

```text
WorkflowWorker namespace concurrent poller limit exceeded
ActivityWorker namespace concurrent poller limit exceeded
```

本次日志统计约：

```text
WorkflowWorker: 77 次
ActivityWorker: 24 次
合计: 101 次
```

【为什么 prod 没问题，YL 有问题】  
prod 环境给 `cgbundle` namespace 配了较高限制：

```yaml
frontend.namespaceCount:
  - constraints: { namespace: 'cgbundle' }
    value: 100000
```

但 YL 环境只给 `default` namespace 配了，`cgbundle` 没配，所以走 Temporal 默认值，容易触发并发 poller 限制。

【验证】  
将 YL 环境 Temporal dynamic config 中 `cgbundle` 的限制补齐

```yaml
frontend.namespaceCount:
- constraints:
    namespace: cgbundle
  value: 100000
```

同步补齐 `frontend.namespaceRPS`、`history/matching/worker.persistenceNamespaceMaxQPS` 中 `cgbundle` 的配置，和 prod 保持一致。

2. 验证服务分发速度，任务速度由原来的 20分钟，降低到30s

3. 变更后观察 `cginstanceagent` 日志中没有出现：

```text
namespace concurrent poller limit exceeded
```

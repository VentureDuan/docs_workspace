# Vault Usage Guide for Agents

本 Vault 用于 `Codex + Obsidian` 的长期记忆。Obsidian 负责沉淀长期上下文，Agent 负责即时推理、加工、整理和复用。

## 目录结构

- `00_Inbox/`：临时输入。
- `01_Projects/`：项目上下文。
- `02_Debugging/`：排障记录。
- `03_Agent_Context/`：给 Agent 看的长期上下文。
- `04_Workflows/`：Prompt、SOP、Checklist、模板。
- `05_Learning/`：学习笔记。
- `99_Archive/`：归档。

大项目必须使用目录形式，例如 `01_Projects/FileDispatch 内容分发平台/`。项目总览写入该目录下的 `项目概览.md`，后续架构、排障、决策、会议、Prompt 等内容继续放在对应项目目录内，不要把大项目文档散落在 `01_Projects/` 顶层。

## 你应该先读这些文件

- `03_Agent_Context/我的技术栈.md`
- `03_Agent_Context/当前项目.md`
- `04_Workflows/代码审查清单.md`
- `04_Workflows/技术方案模板.md`

如果文件不存在，先按现有 Vault 内容工作，不要为了补齐上下文而编造内容。

## 应该记录什么

优先记录：

- 决策过程
- 项目上下文
- 用户自己的技术判断
- Prompt、工作流、模板
- 学习沉淀
- 有后续价值的备忘

不要记录：

- 网上能随时搜到的通用知识
- API 文档的复制粘贴
- 一次性问题的答案
- Agent 能快速重新生成的普通总结
- 没有后续价值的碎片信息

## 写入规则

- 新内容先写到 `00_Inbox/agent-drafts/`。
- 长期上下文草稿可写到 `03_Agent_Context/drafts/`。
- 每次生成内容必须标注来源、假设和待确认项。
- 不要删除旧笔记，只能追加或新建。
- 不要直接修改正式决策记录；如果未来存在 `02_Decisions/`，该目录视为只读。
- 修改已有笔记前，先阅读原文，保留原有上下文、链接和用户表达。
- 新建大项目时，先创建 `01_Projects/<项目名>/项目概览.md`，再按主题补充同目录文档。

## Agent 权限

只读区：

- `01_Projects/`
- `02_Debugging/`

可写草稿区：

- `00_Inbox/agent-drafts/`
- `03_Agent_Context/drafts/`

可追加区：

- `02_Debugging/`
- `04_Workflows/`

禁止区：

- 私人日记
- 密码、密钥
- 财务信息
- 身份信息

## 我的偏好

- 语言：中文为主。
- 技术栈：Go、后端、微服务、MySQL、Redis、MQ。
- 输出风格：先结论，再细节。
- 代码建议：必须说明风险和迁移成本。

## AI 生成笔记模板

默认使用：

```markdown
背景：
问题：
方案：
为什么这么做：
踩坑：
下次怎么复用：
相关链接：

来源：
假设：
待确认项：
```

字段要求：

- `背景：` 记录问题出现的上下文、项目状态、约束条件。
- `问题：` 写清楚真正要解决的矛盾、风险或判断点。
- `方案：` 记录最终选择或当前可执行做法。
- `为什么这么做：` 记录取舍依据、技术判断和替代方案对比。
- `踩坑：` 记录容易遗漏、已经验证失败或未来可能再次遇到的问题。
- `下次怎么复用：` 提炼成可迁移的方法、检查清单、Prompt 或工作流。
- `相关链接：` 只放对未来追溯有帮助的链接。
- `来源：` 说明内容来自用户、当前对话、项目文件还是外部资料。
- `假设：` 明确 Agent 生成内容时依赖的前提。
- `待确认项：` 列出需要用户或后续验证确认的内容。

## 判断标准

写入 Vault 前先问：

- 三个月后是否还有用？
- 是否体现用户自己的判断、取舍或经验？
- 是否能帮助未来的 Agent 更快理解项目或用户偏好？
- 是否能复用为 Prompt、模板、检查清单或工作流？

如果答案大多是否定的，只在对话中回答，不写入 Vault。

## 编码要求

- Markdown、JSON、YAML、文本文件默认使用 UTF-8。
- 在 Windows 环境下处理中文时，不依赖本地默认编码。
- 编辑已有文件时，尽量保留原有换行风格和编码。
